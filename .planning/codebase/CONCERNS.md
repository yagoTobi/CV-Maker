# Concerns & Technical Debt
> Generated by codebase mapper -- 2026-03-29

## Critical

### No Authentication / Authorization
- Issue: `backend/dependencies.py` `get_current_user()` trusts the `X-User-Id` header verbatim, defaulting to `"local"`. Any client can impersonate any user by setting this header.
- Files: `backend/dependencies.py`
- Impact: In a multi-user deployment, any user can read, modify, or delete another user's CV data, versions, and voice profiles. Data isolation is entirely header-trust-based.
- Fix approach: Implement JWT or Cognito token validation in `get_current_user()`. The function is already a FastAPI `Depends()` -- swap the header read for token verification. All routes already use it, so the change is localized.

### No Rate Limiting on AI Endpoints
- Issue: AI-powered endpoints (`/chat/*`, `/tailor/suggest-changes`, `/cv-import`, `/voice/extract-cv`) have no rate limiting. Each call invokes AWS Bedrock (Claude), which costs money per request.
- Files: `backend/routes/chat.py`, `backend/routes/tailor.py`, `backend/routes/cv_import.py`, `backend/routes/voice_interview.py`
- Impact: A single malicious or buggy client can run up significant AWS Bedrock costs. The match-analysis and tailor endpoints are particularly expensive (Sonnet model, 8192 max_tokens).
- Fix approach: Add a FastAPI rate-limiting middleware (e.g., `slowapi`) or per-user request counting. Prioritize the Sonnet-calling endpoints.

### Voice Session Transcripts Stored In-Memory Only
- Issue: Voice interview transcripts are stored in a plain Python dict (`_sessions`) inside the process. No persistence, no multi-process support.
- Files: `backend/routes/voice_interview.py` (line 56: `_sessions: dict[str, dict] = {}`)
- Impact: If the server restarts between a voice interview ending and the `extract-cv` call, the transcript is lost. With multiple workers (uvicorn `--workers N`), a client connected to worker A gets a 404 on worker B.
- Fix approach: Store transcripts in Redis or DynamoDB with TTL. The existing `SESSION_TTL_SECONDS = 3600` pattern maps directly to a TTL attribute.

### Voice Profile Endpoint Accepts Arbitrary Dict
- Issue: `POST /voice/profile` accepts `profile: dict` with no validation -- any JSON payload is saved directly to storage.
- Files: `backend/routes/voice_interview.py` (line 345-352)
- Impact: Allows arbitrary data injection into the storage backend. Could be used to store large payloads (no size limit) or malformed data that breaks voice profile reads downstream.
- Fix approach: Define a Pydantic model for the voice profile schema and validate input.

## Moderate

### In-Memory LLM Cache With No Size Bound
- Issue: `backend/services/llm_cache.py` uses a plain dict `_cache` with TTL eviction but no max-size eviction. Each cached value is a full AI response string (potentially several KB).
- Files: `backend/services/llm_cache.py`
- Impact: Over time, the cache grows unbounded in memory. A high-traffic server making diverse queries could accumulate hundreds of MB of cached AI responses before TTL expires them.
- Fix approach: Add a max-size limit (e.g., LRU with `maxsize`), or switch to `functools.lru_cache` / `cachetools.TTLCache` which handles both TTL and max-size.

### Synchronous File I/O in Async Storage Backend
- Issue: `FileStorage` methods are declared `async` but perform all file I/O synchronously with `open()`, `json.load()`, `json.dump()`, `os.listdir()`, etc. This blocks the event loop.
- Files: `backend/services/file_storage.py` (all methods)
- Impact: Under concurrent requests, file I/O operations block the async event loop, degrading response times for all endpoints. The `list_versions()` method reads every JSON file sequentially.
- Fix approach: Use `aiofiles` for async file I/O, or offload to `asyncio.to_thread()`. The DynamoDB backend has the same issue (sync boto3 calls in async methods) but can be solved with `aioboto3`.

### DynamoDB Backend Uses Synchronous boto3
- Issue: `DynamoStorage` methods are `async` but call `self._table.query()`, `self._table.get_item()`, etc. synchronously.
- Files: `backend/services/dynamo_storage.py`
- Impact: Same event-loop blocking as `FileStorage`. Under load, a single slow DynamoDB query blocks all concurrent requests.
- Fix approach: Use `aioboto3` or wrap calls in `asyncio.to_thread()`.

### Large Component: CVFormBuilder (952 lines)
- Issue: `CVFormBuilder.tsx` is 952 lines with extensive local state, resize logic, tune mode, import handling, and PDF preview management all in one component.
- Files: `frontend/src/features/form-builder/CVFormBuilder.tsx`
- Impact: Difficult to maintain, test, or modify any single concern without risk of breaking others. The component has ~15 `useState` calls and multiple `useEffect`s with complex dependencies.
- Fix approach: Extract the resize panel logic into a `useResizePanel` hook, the PDF preview into a `<PreviewPanel>` component, and the tune-mode section into a separate component. The sidebar nav could also be extracted.

### Large Hook: useFormBuilder (693 lines)
- Issue: `useFormBuilder` returns 40+ functions and values, all wrapped in a single `useMemo`. The dependency array for the `useMemo` is 43 items long.
- Files: `frontend/src/hooks/useFormBuilder.ts`
- Impact: Any change to any callback triggers a new memoized object. The massive return interface makes it hard to understand what subset of functionality any consumer actually needs.
- Fix approach: Split into domain-specific hooks: `useWorkEntries`, `useEducationEntries`, `useSkillCategories`, etc. Each hook manages its own slice of form data and returns a small focused API.

### Dead Code: CVEdit System and parseEditsFromResponse
- Issue: `types/index.ts` contains `CVEdit`, `parseEditsFromResponse()`, `findActualMatch()`, `normalizeWhitespace()`, and `applyEdit()` -- a full text-based edit system for a removed EditorScreen. `useChat.ts` still exports `applyEdit` and `undoEdit` functions that are never called by any component.
- Files: `frontend/src/types/index.ts` (lines 43-100, 249-268), `frontend/src/hooks/useChat.ts` (lines 203-226, 251, 265)
- Impact: Approximately 100 lines of dead code in the type file and additional dead paths in `useChat`. Confuses developers about how edits work (the active system is form-data-based via `useTailor`, not text-based).
- Fix approach: Remove `CVEdit`, `parseEditsFromResponse`, `findActualMatch`, `normalizeWhitespace`, and `applyEdit` from `types/index.ts`. Remove `applyEditToContent`, `undoEdit`, and `editHistory` from `useChat.ts`.

### Duplicate Error Boundary Components
- Issue: Two error boundary implementations exist: `FeatureErrorBoundary` (styled with CSS modules, used in routes) and `ErrorBoundary` (inline styles, used at root). They serve the same purpose with slightly different UIs.
- Files: `frontend/src/components/FeatureErrorBoundary.tsx`, `frontend/src/features/shared/ErrorBoundary.tsx`
- Impact: UI inconsistency between root-level errors and feature-level errors. Maintenance burden of two components doing the same thing.
- Fix approach: Consolidate into a single `ErrorBoundary` component with configurable severity/styling via props.

### Silent Error Swallowing in API Client
- Issue: Many `api.*` methods catch errors and return null/empty values without surfacing the error to the caller (e.g., `fetchTemplates` returns `[]`, `loadUserData` returns `null`, `deleteVersion` returns `false`). The `console.error` calls are the only indication something went wrong.
- Files: `frontend/src/services/api.ts` (lines 90-93, 182-184, 209-211, 274-277, 283-286)
- Impact: UI shows empty/stale state with no feedback to the user. Template loading fails silently; version deletion fails silently; version loading fails silently.
- Fix approach: Bubble errors to callers via thrown exceptions or a `Result<T, Error>` pattern, and let the UI components decide how to display error states.

### Unpinned Backend Dependencies
- Issue: `requirements.txt` uses `>=` for all dependencies with no upper bounds (e.g., `fastapi>=0.115.0`, `boto3>=1.34.0`). No lockfile.
- Files: `backend/requirements.txt`
- Impact: Builds are not reproducible. A new major version of any dependency (particularly `fastapi`, `pydantic`, or `boto3`) could introduce breaking changes. The Dockerfile builds from this file, so production images vary.
- Fix approach: Pin to exact versions or use `~=` (compatible release) with a lockfile (`pip-compile` or `uv`). At minimum, add upper bounds: `fastapi>=0.115.0,<1.0`.

### API_BASE URL Inconsistency Between Files
- Issue: `api.ts` uses `import.meta.env.VITE_API_URL || 'http://localhost:8000/api'` while `useVoiceInterview.ts` uses `import.meta.env.VITE_API_URL || http://${window.location.hostname}:8000/api`. The voice hook also constructs `WS_URL` by regex-replacing `http` with `ws` in the API base.
- Files: `frontend/src/services/api.ts` (line 5), `frontend/src/hooks/useVoiceInterview.ts` (lines 13-14)
- Impact: In non-localhost development environments (e.g., remote dev servers, Docker), the two files may resolve to different backends. The WS URL construction via regex is fragile.
- Fix approach: Centralize the API base URL in a single config module. Derive the WS URL from the same source.

## Low Priority

### Production console.log Statements in VoiceWidget
- Issue: `VoiceWidget.tsx` has 4 `console.log` statements that are NOT gated by `import.meta.env.DEV` (unlike `useVoiceInterview.ts` which correctly gates its logging).
- Files: `frontend/src/features/voice-widget/VoiceWidget.tsx` (lines 31-33, 41)
- Impact: Noisy browser console output in production. Leaks form data field names and counts to anyone with dev tools open.
- Fix approach: Gate with `import.meta.env.DEV` or remove. The `useFormBuilder.ts` log on line 109 is already correctly gated.

### useFormBuilder Sync Log in Development
- Issue: `useFormBuilder.ts` line 109 logs imported data details only in dev mode, which is correct, but includes full form data structure information.
- Files: `frontend/src/hooks/useFormBuilder.ts` (line 109)
- Impact: Minor -- correctly gated by `DEV` flag, but could be removed once the import flow is considered stable.
- Fix approach: Remove or reduce verbosity.

### No Input Validation on LaTeX Content Size
- Issue: The `/compile` endpoint accepts arbitrary `tex_content` with no size limit. A large payload could cause memory issues or long compilation times (90s timeout exists but is generous).
- Files: `backend/routes/compile.py`
- Impact: Low risk in single-user mode, but in a multi-user deployment, one user submitting a very large LaTeX document could tie up the compilation process.
- Fix approach: Add a `max_length` validator on the `tex_content` field in the Pydantic model (e.g., 500KB).

### Stale EditorScreen References in MEMORY.md
- Issue: The project MEMORY.md references `/editor` route and `EditorScreen.tsx` but this route no longer exists in `App.tsx` and the file has been removed. The editor components (`JobInput`, `MatchSummaryBar`, `TailorPanel`) are now used directly in `CVFormBuilder` instead.
- Files: Project MEMORY.md (route structure section)
- Impact: Misleading context for future development sessions.
- Fix approach: Update MEMORY.md to remove `/editor` route references.

### No 404/Not-Found Route
- Issue: `App.tsx` has no catch-all route for undefined paths. Navigating to a non-existent URL shows a blank page (the `Suspense fallback={null}` renders nothing).
- Files: `frontend/src/App.tsx`
- Impact: Poor UX when users navigate to invalid URLs.
- Fix approach: Add a `<Route path="*" element={<NotFoundPage />} />` catch-all.

### axiosInstance.isAxiosError Bug in CV Import
- Issue: Line 312 of `api.ts` calls `axiosInstance.isAxiosError(err)` but `isAxiosError` is a static method on `axios`, not on an axios instance. This would fail at runtime.
- Files: `frontend/src/services/api.ts` (line 312)
- Impact: When `importCV` catches an error with a response body, the `isAxiosError` check silently fails (returns `undefined`/falsy), so the error detail from the server is never extracted. The generic `err.message` is used instead.
- Fix approach: Change to `axios.isAxiosError(err)` (using the default import, not the instance).

### Missing TemplateSelector Error Boundary
- Issue: The `/build` route renders `<TemplateSelector />` without a `FeatureErrorBoundary` wrapper, unlike all other feature routes.
- Files: `frontend/src/App.tsx` (line 22)
- Impact: If the template fetching or rendering throws, the error propagates to the root boundary, which reloads the entire page instead of showing an inline retry UI.
- Fix approach: Wrap with `<FeatureErrorBoundary>` like the other routes.

### Accessibility Gaps
- Issue: Only 8 `aria-*`/`role`/`tabIndex` attributes across the entire frontend codebase. Interactive elements like the resize handle, nav drag items, and tailor suggestion cards lack keyboard accessibility.
- Files: Most `.tsx` files under `frontend/src/features/`
- Impact: Screen reader users and keyboard-only users cannot effectively navigate the form builder sidebar, reorder sections, or interact with tailor suggestion cards.
- Fix approach: Add `role`, `aria-label`, and `tabIndex` to interactive non-button elements. Implement keyboard handlers for drag-and-drop. Add skip navigation links.

## Observations

### Mobile Responsiveness Is Partial
- The form builder has responsive breakpoints at 1100px, 900px, and 600px (`frontend/src/features/form-builder/CVFormBuilder.module.css` lines 947-994) that progressively hide the preview panel and stack the layout.
- The editor components (TailorPanel, MatchSummaryBar) have almost no responsive CSS -- only `JobInput.module.css` has one `@media` rule at 1024px.
- Per MEMORY.md: "Mobile responsiveness for EditorScreen (currently non-functional below 640px)" -- the tune mode within the form builder inherits this problem.

### FileStorage Lacks Concurrency Safety
- `FileStorage.list_versions()` iterates directory contents and reads each file without any locking. `update_version()` does read-modify-write without atomicity.
- Files: `backend/services/file_storage.py`
- Under concurrent writes (even from a single process with async handlers), two simultaneous updates could lose data. Acceptable for single-user local dev but not for any shared deployment.

### LaTeX Compilation Runs as Subprocess Without Sandboxing
- The compiler runs `pdflatex`/`xelatex` via `subprocess.run()` in a temp directory. The dangerous-pattern sanitizer (`LaTeXCompiler.DANGEROUS_PATTERNS`) blocks many known exploit vectors, but the process runs with the same permissions as the backend.
- Files: `backend/services/latex_compiler.py` (lines 26-40, 103-114)
- The sanitizer is regex-based and could potentially be bypassed by obfuscated LaTeX. In a multi-user deployment, consider running compilation in a Docker container or nsjail sandbox.

### Context Architecture Migration Is In-Progress
- `AppContext.tsx` is now a compatibility shim over three domain contexts (`JobContext`, `CVContext`, `ToolsContext`). The comment says "New consumers should prefer useJobContext(), useCVContext(), or useToolsContext() directly" but most components still use `useAppContext()`.
- Files: `frontend/src/contexts/AppContext.tsx`, `frontend/src/features/form-builder/CVFormBuilder.tsx`
- This is intentional tech debt with a clear migration path, but consumers using `useAppContext()` get the merged object which defeats the performance benefit of split contexts.

### Backend Has No Frontend Test Coverage for API Routes
- Backend has solid test coverage for templates (21 rendering + 18 compilation tests) and CV extraction (multiple test files), but no tests for the API route handlers themselves (chat, tailor, cv_versions, compile, user_data).
- Files: `backend/tests/` (all files focus on templates and extraction)
- Route-level bugs (e.g., incorrect Pydantic model definitions, wrong HTTP status codes, missing auth checks) can only be caught by manual testing or frontend integration.
