# Phase 4 -- UI Design Contract

> Visual and interaction contract for drag-and-drop reordering of sections and entries on the web CV. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase adds drag-and-drop reordering to the direct-edit web CV. Users drag sections by their headers and entries by their items using dedicated grip handles in a left gutter. All drag chrome is hover-only, matching the Phase 3 pattern for add/delete/toggle controls. The CV document itself is unchanged -- only the surrounding chrome (grip handles, drop indicators) is new.

The design contract has two layers:
1. **CV Document Layer** -- Unchanged from Phase 2. EB Garamond, LaTeX-derived spacing, black-on-white.
2. **Chrome Layer** -- Grip handles and drop line indicator using existing app design tokens (--accent, --text-secondary).

## Interaction Contract

### Grip Handle States

Every grip handle (section-level and entry-level) follows the Phase 3 hover-only chrome pattern:

| State | Visual Treatment | Cursor |
|-------|-----------------|--------|
| Default (no hover on parent) | Invisible (`opacity: 0`) | N/A (not visible) |
| Parent hovered | Visible (`opacity: 1`), grip icon in `--text-secondary` | `grab` |
| Grip hovered | Grip icon turns `--accent`, background `--bg-hover`, border-radius 4px | `grab` |
| Active drag (mousedown on grip) | Grip icon stays `--accent`. Dragged item at `opacity: 0.4`. Drop line visible. | `grabbing` |
| Drag complete | Grip returns to default. Item opacity restored to 1. Drop line hidden. | `grab` (if still hovering) |

**Transition:** `opacity 0.15s ease` on grip visibility (matches Phase 3 toggle/delete transitions).

**Source:** D-02 (hover-only), D-04 (0.4 opacity + drop line), D-09 (contentEditable-safe pattern).

### Drop Line Indicator

| Property | Value |
|----------|-------|
| Color | `#3B82F6` (var(--accent)) |
| Height | 2px |
| Position | Between items at the target drop position |
| Width | Full width of the section content area, plus 4px overshoot on each side |
| Border-radius | 1px (half of height, for rounded ends) |
| Visibility | Only visible during an active drag, at the current drop target position |
| Transition | None (instant show/hide -- no animation delay during fast dragging) |

**Source:** D-04 (horizontal blue accent line between items).

### Drag Ghost

| Property | Value |
|----------|-------|
| Browser ghost | Suppressed via 1x1 transparent canvas/image set in `onDragStart` |
| Custom ghost | None -- feedback is drop line + opacity change only |

**Source:** D-05 (hide default ghost).

### Drag Scope Rules

| Drag Type | Container | Drop Targets | Data Attribute |
|-----------|-----------|--------------|----------------|
| Section | `[data-drag-section]` on SectionWrapper root | Other `[data-drag-section]` elements | `data-drag-section="{sectionKey}"` |
| Entry | `[data-drag-entry]` on EntryWrapper root | Other `[data-drag-entry]` within same section | `data-drag-entry="{entryIndex}"` |

**Entry isolation:** Entry drag events call `e.stopPropagation()` on all handlers to prevent bubbling to section-level drag handlers. This is the established pattern from `useDrag.ts`.

**Source:** D-06 (entries within own section only), D-07 (section = sectionOrder, entry = array positions), D-09 (data attributes for scoping).

### Single-Entry Sections

Sections with only one entry do NOT show entry-level grip handles. There is nothing to reorder. Section-level grip handles always show (sections can always be reordered relative to other sections, regardless of entry count).

### Keyboard Behavior

No keyboard shortcuts for drag-and-drop in this phase. Drag is mouse/pointer-only. Keyboard accessibility for reordering is deferred to v2 (POLISH-02).

### contentEditable Safety

The proven pattern from `useDrag.ts` applies:
1. No `draggable` attribute in JSX on any element.
2. Grip handle `onMouseDown` sets `draggable = true` on the nearest `[data-drag-section]` or `[data-drag-entry]` ancestor.
3. `onDragEnd` resets `draggable = false` on the element.
4. This prevents the `draggable` attribute from interfering with text cursor positioning in contentEditable fields.

**Source:** D-09, CLAUDE.md key learning #21.

## Layout Integration

### Grip Handle Positioning

**Section-level grip (in SectionWrapper):**

```
[grip] [SECTION HEADER TEXT          ] [eye toggle]
       [section content...                        ]
       [+ Add entry                               ]
```

The grip sits in the left gutter, vertically aligned with the section header baseline. The sectionHeaderRow becomes a 3-column layout: `[gutter] [header] [toggle]`.

**Entry-level grip (in EntryWrapper):**

```
       [grip] [Company Name          Start - End] [X delete]
              [Job Title                Location]
              [* bullet text...                  ]
```

The grip sits to the left of the entry content, vertically aligned with the first line (entry header row). The delete button remains to the right (unchanged from Phase 3).

### Gutter Implementation

The left gutter is implemented via CSS on SectionWrapper and EntryWrapper, NOT as a global page-level column. Each wrapper manages its own grip positioning.

- SectionWrapper: `sectionHeaderRow` grid gains a `gutter-width` column on the left.
- EntryWrapper: Grip is `position: absolute; left: -28px; top: 0;` (gutter-width + gutter-gap from content edge), matching the Phase 3 delete button pattern (`position: absolute; right: -24px`).

### Grid Layout Entries (Skills, Awards)

Skills and awards use CSS subgrid (`entryWrapGrid`). Entry-level grips in grid contexts use `position: absolute; left: -28px;` on the EntryWrapper, consistent with the standard entry grip positioning. The subgrid layout is preserved.

## Accessibility Notes

- Grip handles use `<button>` elements with descriptive `aria-label` values.
- Drop line indicator has `aria-hidden="true"` (visual-only feedback, not announced).
- No keyboard shortcuts for reordering in this phase. Keyboard reordering is deferred to v2 (POLISH-02).
- Grip handles are focusable via Tab (native button behavior) but drag requires mouse.
- `cursor: grab` / `cursor: grabbing` provides visual affordance for drag capability.
- Drag operations are non-destructive; no confirmation or undo mechanism required.
